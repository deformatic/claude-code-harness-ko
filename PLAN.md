# 한국어 버전 제작 계획

## 목표

- `MAP.md`에 정의된 책의 구조, 학습 경로, 핵심 메시지를 유지한 채 한국어 독자가 빠르게 이해할 수 있는 한국어 버전을 만든다.
- 결과물은 "직역본"이 아니라 "원문의 정보 구조를 보존한 이해하기 쉬운 한국어 기술 문서"를 목표로 한다.
- 작업 방식은 Subagent 분업을 전제로 설계하되, 최종 용어 통일과 품질 책임은 메인 에이전트가 가진다.

## 범위 해석

- 현재 확인 가능한 기준 문서는 `MAP.md`이며, 여기에는 다음 정보가 포함된다.
- 전체 책 구조: Preface + 7개 Part + Appendix
- 총 30개 본문 챕터 + 7개 Appendix
- 독자별 추천 읽기 경로 3종
- Chapter 3(Agent Loop)를 중심축으로 삼는 지식 맵
- 서술 규칙: 소스 참조, 근거 수준, Mermaid, 인터랙티브 시각화
- 따라서 한국어판은 최소한 다음 층위를 유지해야 한다.
- 제목 체계
- 파트/챕터 간 관계
- 추천 읽기 경로
- 용어 기준
- 근거 수준 표기 방식

## 작업 원칙

- 제목은 가능한 한 원문의 개념을 보존하되, 한국어 독자가 한 번에 의미를 파악할 수 있게 다듬는다.
- 본문은 직역보다 의미 단위 재서술을 우선한다.
- Claude Code, Agent Loop, Prompt Cache 같은 핵심 고유 개념은 무리하게 번역하지 않고, 필요 시 한국어 병기를 붙인다.
- 예시: `Agent Loop(에이전트 루프)`, `Prompt Cache(프롬프트 캐시)`
- 독자 안내 문구와 메타 설명은 한국어 문장 리듬에 맞춰 재작성한다.
- 장별 독립성을 유지하되, 책 전체에서 같은 개념은 항상 같은 번역을 사용한다.
- 링크 처리 원칙은 URL 번역과 동일하지 않다. `MAP.md`의 링크는 원문 URL 기준으로 수집하고, 한국어화는 링크 텍스트, 제목, 부연 설명에서 수행한다.
- URL은 기본적으로 보존한다. Vercel 배포에서는 다음 원칙으로 유지/변환한다.
- 외부 URL(`https://`, `http://`)은 번역하지 않고 그대로 둔다.
- 내부 URL에서 `/en/` 경로는 `/ko/`로 매핑한다.
- 앵커(`#section`)와 파일명(`.html`)은 가능하면 유지한다.
- 도메인 변경이 필요한 경우에만 별도 리디렉트 규칙으로 처리한다.

## 산출물 구조

- `PLAN.md`: 전체 실행 계획과 운영 규칙
- `GLOSSARY.md`: 핵심 용어 영한 기준표
- `STYLEGUIDE.md`: 문체, 제목 번역 규칙, 표기 규칙
- `ko/preface.md`: 한국어 서문
- `ko/part1/` ~ `ko/part7/`: 파트별 한국어 챕터
- `ko/appendix/`: 부록 한국어 버전
- `ko/README.md`: 추천 읽기 경로와 전체 안내
- 배치별 검수 로그 및 이슈 목록

## Subagent 운영 구조

- 메인 에이전트
- 역할: 전체 구조 관리, 용어 승인, 문체 통일, 최종 병합, 품질 검수
- 번역 Subagent
- 역할: 지정된 배치의 한국어 초안 작성
- 리뷰 Subagent
- 역할: 소스/번역 대조, 용어 일관성 확인, 한국어 가독성 점검
- 각 Subagent에는 고정 패킷을 넘긴다.
- 포함 항목: 소스 범위, 챕터 브리프, 스타일 가이드, 용어집, 금지 용어 변형, 출력 경로 규칙
- 각 Subagent는 챕터마다 3개 결과를 반환한다.
- 한국어 번역 초안
- 새 용어 제안 또는 용어 차이 목록
- 해석이 애매한 지점과 검수 메모
- 핵심 기술 용어의 새 번역안은 메인 에이전트 승인 전까지 다른 배치에서 재사용하지 않는다.

## 작업 순서

### 1. 기준선 확정

- `MAP.md`를 기준으로 전체 목차와 장 번호를 한국어판용으로 고정한다.
- 번역하지 않을 고유명사 목록을 먼저 만든다.
- Chapter 3가 책의 중심축이라는 설명을 한국어판 전체 구조 설명에도 동일하게 반영한다.

### 2. 용어집 선행 작성

- 우선 확정해야 할 핵심 용어 후보:
- Harness Engineering
- AI Coding Agent
- Tool System
- Agent Loop
- Prompt Engineering
- Context Management
- Prompt Cache
- Safety & Permissions
- Agent Spawning
- Hooks
- Sandbox
- Compaction
- Token Budgeting
- Feature Flags
- 각 용어마다 다음 기준을 기록한다.
- 영문 원어
- 권장 한국어 표기
- 첫 등장 시 병기 형태
- 번역하지 않는 이유 또는 번역 선택 이유
- 금지할 변형 표현
- 용어집의 우선 소스는 다음과 같이 둔다.
- `MAP.md`의 장/파트 제목
- Preface와 Reading Preparation의 반복 표현
- Appendix C Glossary
- 내비게이션 및 UI 문자열

### 3. 스타일 가이드 고정

- 제목 번역 규칙을 정한다.
- 원제 의미 유지
- 필요 시 부제에서 설명 보강
- 숫자/장 번호 표기를 일관되게 유지
- 문체 규칙을 정한다.
- 설명형 문장 우선
- 과도한 구어체 금지
- 기술 개념은 짧고 명확한 정의를 먼저 제시
- 표기 규칙을 정한다.
- 코드/경로/변수명은 원문 유지
- 링크 텍스트는 가능하면 한국어화
- 근거 수준은 고정 번역 사용
- 예시:
- `source evidence` -> `소스 근거`
- `reverse engineering` -> `리버스 엔지니어링 근거`
- `Inference` -> `추론`

### 4. 목차와 안내 문서 먼저 한국어화

- Preface와 Reading Preparation을 가장 먼저 만든다.
- 동시에 Chapter 3와 Appendix C Glossary를 초기 기준선으로 잡는다.
- 이유:
- 책 전체 톤과 독자 기대치를 먼저 고정할 수 있다.
- 이후 챕터 번역 시 동일한 설명 프레임을 재사용할 수 있다.
- 추천 읽기 경로 3종은 한국어 독자 기준으로 문장을 더 직관적으로 다듬는다.
- Chapter 3는 책 전체의 중심축이므로 이 장의 번역이 전체 서술 어조와 핵심 용어를 고정하는 기준점이 된다.

### 5. 챕터 번역 배치

- 다음 묶음으로 병렬 처리한다.
- Batch 0: Preface + Reading Preparation + Chapter 3 + Appendix C Glossary
- Batch 1: Part I `Ch1-Ch4b`
- Batch 2: Part II `Ch5-Ch8`
- Batch 3: Part III `Ch9-Ch12` + Part IV `Ch13-Ch15`
- Batch 4: Part V `Ch16-Ch19`
- Batch 5: Part VI `Ch20-Ch24`
- Batch 6: Part VII `Ch25-Ch30`
- Batch 7: Appendix `A, B, D, E, F, G`
- 상태:
- Batch 1 완료 (`ko/part1-ch01.md` ~ `ko/part1-ch04b.md`)
- Batch 2 완료 (`ko/part2-ch05.md` ~ `ko/part2-ch08.md`, `ch06b` 포함)
- Batch 3 완료 (`ko/part3-ch09.md` ~ `ko/part4-ch15.md`)
- Batch 4 완료 (`ko/part5-ch16.md` ~ `ko/part5-ch19.md`)
- Batch 5 완료 (`ko/part6-ch20.md` ~ `ko/part6-ch24.md`)
- Batch 6 완료 (`ko/part7-ch25.md` ~ `ko/part7-ch30.md`)
- Batch 7 완료 (`ko/appendix-a-file-index.md` ~ `ko/appendix-g-auth-subscription.md`)
- 배치 기준:
- Batch 0은 문체와 용어를 고정하기 위한 기준선 배치다.
- Part I은 구조와 제어 흐름 어휘의 기준이 되므로 초반 우선순위가 높다.
- Part III와 Part IV는 context/cache 최적화 어휘가 강하게 겹치므로 같은 검수 레인에서 처리한다.
- Part V는 safety/permissions/security 표현의 일관성이 중요하므로 묶어서 검수한다.
- Part VI는 subagent, teams, skills, plugins, memory 등 확장 하위시스템이 몰려 있어 한 배치로 다룬다.
- Part VII와 나머지 Appendix는 앞선 용어가 충분히 안정된 뒤 처리한다.

### 6. 각 Subagent의 공통 작업 단위

- 맡은 범위의 원문 구조를 보존한다.
- 제목 초안 작성
- 문단 단위 한국어 재서술
- 번역이 애매한 용어 후보 표시
- 원문 링크/교차참조 유지
- 결과 제출 형식:
- 요약
- 번역본 초안
- 미결 용어 목록
- 검수 필요 포인트

### 7. 메인 에이전트 병합

- Subagent 결과를 파트 단위로 병합한다.
- 중복 번역 또는 상충하는 용어를 정리한다.
- 추천 읽기 경로와 장 번호 연결이 정확한지 재확인한다.
- 파트 설명과 챕터 제목 사이의 톤 차이를 줄인다.

### 8. 품질 검수

- 1차 검수: 구조 검수
- 장 번호, 파트 번호, 링크, 파일 경로 확인
- 2차 검수: 용어 검수
- 동일 개념이 다른 한국어로 번역되지 않았는지 확인
- 3차 검수: 가독성 검수
- 한국어 문장이 설명형으로 자연스러운지 확인
- 4차 검수: 기술 정확성 검수
- 의미가 바뀌거나 축약되면서 기술적 오해를 만들지 않았는지 확인
- 배치 승인 조건:
- 누락된 섹션이 없어야 한다.
- Markdown 구조가 깨지지 않아야 한다.
- 링크, 이미지 경로, Mermaid, 코드 펜스가 보존되어야 한다.
- 기술 주장과 근거 수준 표기가 바뀌지 않아야 한다.
- 잠금된 핵심 용어에 승인되지 않은 변형이 없어야 한다.

## 검수 체크리스트

- 제목이 원문 의미를 훼손하지 않는가
- 장 번호와 파트 번호가 원문과 일치하는가
- 추천 읽기 경로의 챕터 참조가 정확한가
- Chapter 3 중심축 설명이 일관되게 유지되는가
- 소스 참조, 근거 수준, Mermaid 관련 표기가 일관적인가
- Claude Code 관련 고유명사가 임의로 번역되지 않았는가
- 같은 용어가 챕터마다 다르게 번역되지 않았는가
- 한국어 문장이 지나치게 직역투이지 않은가

## 우선 실행 순서

2. `STYLEGUIDE.md` 초안 작성

## 완료 기준

- 한국어 독자가 `MAP.md` 수준의 구조 정보를 한국어만으로 이해할 수 있다.
- 추천 읽기 경로, 지식 맵, 핵심 파트 구성이 자연스럽게 이어진다.
- 핵심 용어 번역이 문서 전체에서 일관된다.
- Subagent가 작성한 초안들을 병합해도 문체 차이가 크지 않다.
- 이후 장별 본문 확장 작업에 바로 들어갈 수 있을 정도로 운영 기준이 고정된다.
